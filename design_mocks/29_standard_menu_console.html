VoidYield menu proposal / standard operations console

One menu language for every facility

This mock turns the eclectic building menus into a single UI grammar: every facility opens as an operations console with the same anatomy, the same action placement, the same status logic, and the same reading order. The facility changes. The shell does not.

Design principles
  • Top tells you what this building is, what state it is in, and why you should care right now.
  • Center column is always the main interaction area: trade rows, drone roster, recipe queue, or buffers.
  • Right column is always live telemetry and consequences, so players can make decisions without hunting.
  • Bottom command bar always holds primary action, safe secondary actions, and close/help affordances.
What this fixes
  • Marketplace, drone panels, and production panels stop feeling like three different games.
  • Players learn one set of visual rules instead of re-learning every building.
  • Important state moves above the fold: power, throughput, queue health, stock pressure, and blockers.
  • The style keeps VoidYield’s amber, navy, and teal identity without sliding into generic sci-fi UI.

Shared anatomy

1. Hero header Name, state, why-it-matters summary, and four critical metrics.
2. Main interaction block List, roster, recipe queue, or sell ledger always occupies the left column.
3. Telemetry rail Buffers, quality, line health, and route consequences always occupy the right column.
4. Command rail Primary action right-aligned, close/help stable, no hunting for the "do it" button.

Rules to carry into implementation

Status first Every menu should explain its blocker or opportunity before showing controls.
One accent per facility Amber shell stays global; teal, gold, mint, or orange can specialize a facility without breaking the system.
Management before shopping If a menu both controls and purchases, control comes first and buying becomes secondary.
Action phrasing stays human Buttons should say what happens now: sell stock, clear output route, commit recipe, purchase heavy miner.
Review intent: this is a UX proposal, not final implementation. If this direction feels right, the next pass would be to translate the shared shell into reusable CSS/HTML structure for the in-game overlays and then migrate each panel into it without losing its gameplay-specific information.
